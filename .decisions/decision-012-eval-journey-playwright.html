Decision 012: eval-journey Playwright Strategy

Date: 2026-07-07

Status: Decided

Context: eval-journey uses Playwright in x-ray mode for Phase A AC verification. The Jul 7 batch run showed 40-49s per prototype spent on Playwright that confirmed what source analysis already knew (0 fix loops triggered by visual-only failures).

Options

OptionTime CostVisual ProofRisk

Decision: Option A — Keep Playwright (current behavior)

Keep Playwright as the visual truth source for Phase A verdicts.

Rationale

  • Ghost elements are real: RHAISTRAT-432 had DOM rows with zero height (CSS rendering issue). Only Playwright screenshots caught this — source code showed the feature "existed."
  • Phase A screenshots feed the report: The evaluation-report.html Evidence Viewer relies on per-step screenshots for reviewer trust. Removing them degrades the report's value as a communication artifact.
  • 40-49s is acceptable: Total pipeline is 2.5-6 min. Playwright is 15-25% — not the bottleneck. Extract and usability take more time.
  • Fix loop needs screenshots: When eval-fix applies changes, the N+1 capture confirms the fix visually. Code-only would require trusting that the fix compiles = works.
  • Future optimization via Option C: If pipeline time becomes a problem (>10 min), revisit as a targeted optimization. For now, correctness over speed.

Consequences

  • No changes to eval-journey Playwright logic
  • NEEDS_REBUILD logic stays in eval-iterate
  • Screenshot rules (item 3) still apply to Phase A captures
  • Locator resilience (item 9) improves Playwright reliability within the current model

Revisit Trigger

If pipeline time exceeds 10 min consistently, or if Phase A Playwright catches 0 visual-only failures across 50+ runs, reconsider Option C.
