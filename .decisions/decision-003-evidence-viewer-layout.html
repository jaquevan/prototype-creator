Evidence Viewer Layout

Interaction

This is the interactive modal that replaces "View Walkthrough." When a designer clicks "View Evidence" from the AC table, this opens. It needs to show: an AC filter bar, a persona-organized tree of walkthrough steps, and a detail panel with screenshots + think-aloud text. The question is how these three elements are physically arranged.

The classic IDE/file-explorer pattern. Left sidebar shows the persona tree with step nodes (orange dots for evidence). Right panel shows the selected step's screenshot and think-aloud detail. AC filter chips live in a top bar above both panels. I recommend this because it's the most natural mapping to Zack's spec — tree on left, detail on right, filter across the top.

Works well when

  • Multiple personas with 8-10 steps each — tree stays navigable
  • Familiar pattern (VS Code, Finder) — low learning curve
  • Persistent tree lets you jump between any step instantly

Watch out for

  • Sidebar takes horizontal space from the screenshot
  • On narrow viewports, sidebar may need to collapse

Personas switch via tabs at the top. Steps are a horizontal dot sequence (like a progress bar) with orange highlights for evidence. The detail panel lives below the stepper. Full-width layout gives the screenshot maximum space. Feels like stepping through a photo gallery.

Works well when

  • Screenshots need maximum width (full modal width)
  • Linear step-through is the primary use case
  • Fewer personas (2-3) — tabs work well

Watch out for

  • 10+ steps overflow horizontally — needs scroll
  • Can't see both personas' trees simultaneously
  • Less spatial overview than a vertical tree

A horizontal split: screenshot takes 60% left, think-aloud context takes 40% right. No persistent tree — navigation is purely sequential (prev/next). Persona and AC selection happens via the top bar. The tree is replaced by a "Step N of M" indicator. Prioritizes immersion in each step over overview.

Works well when

  • Detailed reading of each step's think-aloud is the goal
  • Large screenshots need significant viewport space
  • Linear walkthrough (like watching a replay)

Watch out for

  • No tree overview — can't see which steps have evidence at a glance
  • Jumping to a specific step requires sequential nav
  • Doesn't match the "evidence tree" spec from the ticket

Not a modal at all — the evidence viewer takes over the entire page, with a "Back to Report" link in the header. Same sidebar tree + main panel layout as Option A, but with maximum viewport real estate. The screenshot can be truly full-size. Browser back button returns to the report.

Works well when

  • Evidence review is a serious task, not a quick glance
  • Maximum screenshot fidelity needed
  • Long exploration sessions with many steps

Watch out for

  • Loses report context — can't see AC table while viewing evidence
  • Feels like a separate tool rather than part of the report
  • Self-contained HTML can't use URL routing (no back button)
