Three labeled sections stacked vertically, each with a section header and paragraph of content. Simple, predictable, scannable. Each section is self-contained — the designer reads top to bottom. Familiar from documentation patterns.

Works well when

  • Report has longer narrative content per section
  • Designers read sequentially top-to-bottom
  • Mobile/narrow viewport is important

Watch out for

  • Pushes the status grid and AC table further down
  • Three stacked sections can feel heavy/repetitive

A single cohesive card at the top. The Jira summary is the leading narrative (1-2 sentences), with MR changes and pipeline metadata tucked into a compact 2-column sub-grid below. Everything is in one visual container — the designer sees the full picture in one glance without scrolling. I recommend this because it keeps the executive context tight and gets the user to the status grid faster.

Works well when

  • Status grid and AC table need to be visible without scrolling
  • Summary is brief (1-2 sentences)
  • Designers want to orient fast, not read deeply

Watch out for

  • Less room for narrative if Jira summary is complex
  • Dense — requires careful spacing to avoid feeling cramped

The three pieces of context share a single container, accessed via horizontal tabs. Only one is visible at a time — the designer picks which context to read. This keeps the vertical footprint tiny (only the active tab's content shows) at the cost of hiding two-thirds of the information behind clicks.

Works well when

  • Vertical space is at a premium
  • Each section has significant depth to explore
  • Designers don't always need all three pieces

Watch out for

  • Hides context behind clicks — violates "immediately contextualize"
  • Designer has to remember to check all tabs
  • Three clicks to see everything vs zero in other options

Timeline Narrative

RHAISTRAT-1536 Model Registry RBAC
Ticket: RBAC controls for model registry access management
Built: 14 files, 3 routes (RBAC table, role selector, permission modal)
Pipeline: Medium fidelity, decide mode, 4 decisions made, 2 fix iterations
Result: 4/5 ACs pass, 1 flagged for human review
A chronological story told as a vertical timeline — what was the input (ticket), what was built (MR), how it was built (pipeline), what's the verdict (result). Reads like a narrative progression, ending with the current state. Adds a fourth element (result) that bridges into the status grid below.

Works well when

  • Designer wants to understand the journey, not just the facts
  • The "how we got here" context matters for interpreting results
  • Natural transition into the status grid below

Watch out for

  • Harder to scan — requires reading each line
  • Timeline metaphor feels odd when pipeline ran in minutes
  • Less conventional for report headers
