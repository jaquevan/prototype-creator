Decision 6 of 7

Review Entry & Interface

Interaction

When a designer sees flagged or failed items, they need to enter a review flow — add notes, override verdicts, and ultimately trigger action. The question is: where does the review happen, and what does the interface look like? Currently it's a small modal per flagged item with a textarea. We can do better.

Respond with: "Option B" · "Option A but [change]" · "more options"

Option A
Chosen

Inline Expansion

1Click flagged badge on AC row
↓
2Row expands inline with textarea + verdict toggle
↓
3Type notes, pick override verdict
↓
4Click "Save" — row collapses, badge updates
No modal at all. Clicking a flagged badge expands that AC row inline — a textarea + verdict override buttons appear directly below the criterion text. Save collapses it back. The review stays in context of the AC table without ever leaving the page flow.

Works well when

  • Designer reviews items one at a time in context
  • No modal fatigue — stays in the flow
  • Can see adjacent ACs while reviewing

Watch out for

  • Table gets tall with multiple expanded rows
  • No overview of all reviews in progress

Option B
Recommended Chosen

Review Panel (Slide-out)

1Click "Review N items" CTA or flagged badge
↓
2Right panel slides in (report shifts left)
↓
3Panel shows item list + detail for selected item
↓
4Override verdict, add notes, view evidence inline
↓
5Action bar: Download CSV / Re-run eval
A persistent right panel slides in, showing all items needing review as a vertical list. Selecting one shows its full context: the AC text, evidence screenshot, AI rationale, and a textarea for notes. You can override the verdict (PASS/FAIL), view evidence from the evidence viewer, and when done, hit action buttons at the bottom. I recommend this because it keeps all review items visible as a queue while giving each one full space for context and notes.

Works well when

  • Multiple items to review — queue is visible
  • Evidence context available inline (link to evidence viewer)
  • Report stays visible alongside the panel
  • Natural place for action buttons (bottom of panel)

Watch out for

  • Report width shrinks when panel is open
  • More complex implementation than inline or modal

Option C
Chosen

Enhanced Modal (Current + Actions)

1Click flagged badge or "Review" CTA
↓
2Modal opens with item context + evidence
↓
3Override verdict, add notes in textarea
↓
4Footer: "Download Updated CSV" + "Re-run Iteration"
↓
5Prev/Next to move between flagged items
Keep the existing modal pattern but enhance it: add the evidence screenshot inline, the AI's rationale for why it was flagged, verdict override buttons (PASS/FAIL), and critically — action buttons in the footer. Prev/Next to cycle through all items needing review without closing the modal.

Works well when

  • Familiar pattern — already exists, just enhanced
  • Focused attention on one item at a time
  • Less implementation work (enhance vs rebuild)

Watch out for

  • Modal blocks the report — can't reference other sections
  • No queue overview (prev/next is blind navigation)

Option D
Chosen

Dedicated Review Page

1Click "Review N items" — page view switches
↓
2Full-page review interface: list + detail split
↓
3Each item: evidence, rationale, verdict, notes
↓
4Mark items as reviewed (green checkmark)
↓
5"All reviewed" → Download CSV / Re-run / Back to report
A full-page takeover dedicated to the review task. Left: list of all items needing review (with status badges). Right: full detail view with evidence, notes, and verdict controls. Once all items are reviewed, the action bar appears at the bottom. Like a dedicated "review mode" you enter and exit.

Works well when

  • Many items to review (5+) — queue management matters
  • Maximum space for evidence and notes
  • Clear "done" state when all items reviewed

Watch out for

  • Loses report context entirely
  • Heavy implementation — essentially a second UI
  • Overkill for 1-2 flagged items
